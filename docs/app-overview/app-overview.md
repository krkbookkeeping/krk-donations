What you are building is a fully functional, CRA-aware donor management and charitable receipting system that combines elements of a lightweight CRM, an accounting-grade transaction engine, and a reporting platform. At its core, the system is designed to track donors—both individuals and organizations—store their complete contact information, and maintain a clean, searchable, and maintainable database of those donors. Each donor profile contains full identifying and contact details such as first and last name, organization name where applicable, email, phone number, and full mailing address. It also includes internal notes, a preferred contact method, and a status field that allows donors to be marked as active or archived without ever deleting them. The system must support fast and flexible search functionality, allowing users to locate donors using first name, last name, email, phone number, address, or a general text search that scans across multiple fields. This ensures usability even as the database grows.

The system is designed to support multiple companies under a single user account. Each company is a completely independent data environment with its own donors, donations, categories, payment methods, receipts, and receipt numbering sequence. A user linked to more than one company can switch between them from within the app using a company switcher in the top navigation bar, without signing out. Switching company context reloads all data for the selected company and the active choice is remembered across sessions. There is no cross-company data sharing — every donor record, transaction, and receipt belongs exclusively to one company. This structure supports scenarios such as managing two separate registered charities or operating distinct programs that must maintain separate books and receipt sequences.

A critical capability of the system is the ability to handle duplicate donor records in a controlled and auditable way. Duplicate detection should warn the user when a new donor is being created with the same email or a similar name and phone combination. When duplicates are discovered, the system must provide a structured merge process that allows the user to select a primary donor record and merge one or more secondary records into it. During this process, the user must be able to review and resolve conflicting data fields, ensuring that the correct contact information is retained. All historical donations from the secondary record must be reassigned to the primary record, and the duplicate record should then be archived rather than deleted. Every merge action should be logged to maintain an audit trail, ensuring transparency and traceability.

The donation tracking system itself is built around a key architectural decision: each donation is a single transaction that can be split into multiple allocations. This is essential for real-world use cases, particularly where a single payment contains both receiptable and non-receiptable components, such as event tickets where part of the payment represents an advantage. Each donation record captures the overall transaction details, including the donor, the date, the total amount, the payment method, any reference number such as a cheque or transaction ID, and internal notes. Importantly, the donation record does not store category information directly. Instead, categories are assigned at the allocation level.

The allocation system is the core engine that makes the app function correctly. Each donation can have one or more allocation records, with each allocation representing a portion of the total donation assigned to a specific category. Each allocation includes a category, an amount, and a receiptable status that is automatically derived from the category. The system must enforce strict validation to ensure that the sum of all allocations exactly matches the total donation amount. Users should not be allowed to save a donation if the allocations do not balance, and the interface should clearly display the allocated total versus the donation total to prevent errors. This structure enables accurate handling of scenarios such as splitting a $100 payment into $60 receiptable donation and $40 non-receiptable event ticket revenue.

Categories themselves are fully customizable and managed in a dedicated section of the system. Each category has a name, a designation indicating whether it is tax receiptable, and an active or inactive status. Users must be able to create new categories on the fly while entering donations, ensuring flexibility during real-time data entry. Categories cannot be deleted if they are in use, but they can be archived to prevent further use while preserving historical data integrity. This approach ensures both flexibility and consistency in reporting and receipting.

Payment methods are similarly customizable and managed through their own table. The system should allow users to define, edit, and deactivate payment methods such as cash, cheque, e-transfer, debit, credit card, online payments, and integrations like Square. These payment methods must be selectable during donation entry and available as filters in reporting. The ability to modify and expand payment methods ensures the system can adapt to evolving payment channels without requiring structural changes.

The donation entry experience is one of the most important aspects of the system and must be fast, intuitive, and error-resistant. Users should be able to search for an existing donor or create a new one inline. They then enter the total donation amount and proceed to allocate that amount across one or more categories using a grid-style interface. The system should allow users to add or remove allocation lines easily, automatically populate receiptable status based on category, and display a running total of allocated amounts. Additional fields such as payment method, reference number, and notes complete the entry. The system should also support batch entry scenarios, allowing multiple donations to be entered efficiently for events or weekly collections.

The donations list or transaction detail screen serves as the primary working interface for reviewing and managing donation data. This screen must support multi-criteria filtering, allowing users to refine results using a combination of filters such as date range or specific year, multiple categories, multiple payment methods, specific donors, and receiptable status. These filters must work together using logical AND conditions, narrowing the dataset to only those records that meet all selected criteria. Category and payment method filters should support multi-selection, while donor selection is typically single-select. A clear “reset filters” option should be available, and filtering should ideally update results dynamically for a smooth user experience.

Because of the allocation-based structure, filtering must be aware of allocation-level data. For example, if a donation is split between receiptable and non-receiptable components, and the user filters for receiptable donations, that donation should still appear because part of it qualifies. In the MVP version, the system can display the full donation while applying filters based on its allocations, with the option to expand to allocation-level display in future versions. The transaction screen must also support printing and exporting, allowing users to generate clean, formatted outputs or export data to CSV for further use in accounting systems.

Reporting capabilities include a yearly summary report that aggregates total and receiptable donations per donor, as well as a detailed transaction report that mirrors the transaction screen with filtering capabilities. Additional reporting options such as monthly summaries or category-based reports can be added later, but the MVP should focus on these core reports. All reports should allow filtering by date range, category, payment method, and donor, ensuring flexibility in analysis.

The charitable receipting system is a critical component and must be designed with CRA compliance in mind. Receipts are generated on a yearly basis per donor and must include only the portions of donations that are designated as receiptable through their allocations. Each receipt must include the donor’s name and address, the total eligible donation amount, the year, a unique receipt number, the issue date, and the organization’s name and registered charity number. The system must generate receipts as PDF documents and store a snapshot of the data used to create them.

A crucial rule is that once a receipt has been issued, the underlying donation and allocation data must be locked to prevent changes that would invalidate the receipt. If corrections are needed, the system must support a void-and-reissue process rather than allowing direct edits. Receipt numbers must be unique, sequential, and permanently assigned once generated to ensure compliance and auditability. The system should support both individual and bulk receipt generation, allowing all receipts for a given year to be created and optionally emailed in a single action.

Receipt customization is an important feature that allows organizations to tailor the appearance and wording of their receipts. The system should support editable templates that include customizable headers, body text with merge fields, and footers, along with the ability to include a logo and signature. While the MVP can begin with a simple default template, the architecture should support future expansion to more advanced customization.

Email functionality should allow receipts to be sent directly to donors as PDF attachments, using a customizable email template for subject and body content. The system should track whether a receipt has been sent and when. While basic email functionality is sufficient for the MVP, future enhancements may include integration with dedicated email delivery services to improve reliability and deliverability.

Data integrity and system reliability are reinforced through several supporting features. The system should implement soft deletion, allowing records to be archived rather than permanently removed. An audit trail should track key actions such as creation, modification, and merging of records, even if only at a basic level in the MVP. Data export capabilities should allow users to download donors, donations, and reports in CSV format for backup or integration with accounting systems. Import functionality, at least for donors, should be available to facilitate onboarding of existing data.

A Google Sheets prototype of this system serves as a practical testing environment, mirroring the structure of donors, donations, allocations, categories, and payment methods. This prototype allows for real-world testing of workflows, validation of allocation balancing, filtering of transactions, and calculation of receiptable totals. It also highlights manual processes such as donor merging and receipt generation that will later be automated in the full application. Using this prototype helps identify practical needs, user behaviors, and potential friction points before committing to a full build.

Throughout the design, several potential risks were identified and addressed. These include ensuring receipt data integrity through locking or snapshotting, enforcing strict validation of allocation totals, managing duplicate donors through detection and merging, maintaining proper receipt numbering to meet compliance requirements, and avoiding overbuilding by focusing on a tight, functional MVP. The guiding principle is to prioritize accuracy, simplicity, and reliability over excessive features in the initial version.

In its final form, this system is not just a donation tracker but a robust, compliant, and adaptable platform that supports the full lifecycle of donor management, donation recording, receipting, and reporting. It is designed to handle real-world complexity while remaining practical and efficient to use, with a clear path for future enhancements as needs evolve.
